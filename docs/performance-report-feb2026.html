Studio Changes

GenLayer Studio PRs deployed during this window. 8 releases in 48 hours — from v0.97.4 to v0.99.1.

VersionPRChangeCategoryImpact
v0.97.5 #1432 Pre-decode contract state to eliminate per-read base64 overhead perf GRVT main: 207→193s. Non-sharded: 143→70s (2x). Biggest single win for large-state contracts.
#1428 Cache GenVM precompile artifacts across restarts perf Cold-start time ↓ ~2.5 min. No per-tx impact.
#1431 Permit starvation guard — event-loop yield between permits stability Prevented event-loop blocking during permit batches.
#1424 LLM voting consensus fix stability Correct validator vote aggregation.
v0.98.0 #1434 Upgrade GenVM to v0.2.12, persist execution stats instrumentation Enabled host_handling_time_ms, call_counts, by_method_ms telemetry.
v0.98.1 #1435 Prevent validator thrashing — config hash comparison before reinit stability Eliminated unnecessary validator restarts on config checks.
#1429 OpenRouter preset configurations devex Simplified LLM provider setup.
v0.98.2 #1437 Fix event-loop starvation + ORM session leak stability Pod restarts: 134→0 in 24h. No per-tx speedup, but sustained throughput.
v0.99.0 #1439 Worker error retry with bounded concurrency stability Graceful recovery from transient GenVM failures.
v0.99.1 — Shared decoded value cache across validators perf Validators reuse leader's decoded storage values. No re-decode per validator.

— Validator execution timeout stability Prevents stuck validators from blocking consensus indefinitely.

Contract Changes

CampaignIC v0.4.2 / CampaignShard v0.5.2 — deployed Feb 11 ~23:50 UTC. Focused on eliminating redundant storage reads.

AreaChangeImpact
Cross-contract read amplification Reduced from O(submissions) to O(unique shard+mission groups) Main GRVT reads: 266K → 49K (5.4x reduction). Biggest single win.
Batched shard data reuse Single fetch reused across submit_tweet, resubmit_tweet, get_submissions, etc. Eliminated redundant cross-contract calls within same tx.
Selection algorithm Single-pass selection replacing list/sort patterns CPU savings on large submission sets. O(n) → O(n) with fewer allocations.
Distribution math Fast paths for single-participant and alpha==1 cases Skip expensive math for common degenerate cases.
Shard lookups Single-pass lookup optimizations Shard leader time: marginal improvement (~55→42s).

Combined effect: GRVT main processing 205→31s (6.6x). Rally Beta 148→25s (5.9x). Only affects sharded campaigns with significant state; small-state campaigns (argue.fun, BceE) unchanged.

Methodology

Data Source

All metrics from Rally PRD PostgreSQL (rally-studio-prd namespace, genlayer_state database).

Key Fields

  • consensus_data::jsonb->'leader_receipt'->0->>'processing_time' — total leader execution time (ms)
  • consensus_data::jsonb->'leader_receipt'->0->'execution_stats'->>'host_handling_time_ms' — pure storage I/O time
  • consensus_data::jsonb->'leader_receipt'->0->'execution_stats'->'call_counts'->>'STORAGE_READ' — number of storage read calls
  • timestamp_awaiting_finalization — bigint Unix epoch when tx entered FINALIZED
  • triggered_by_hash IS NOT NULL — distinguishes shard txs from main txs

Phase Boundaries

Phase 0 (v0.97.4)Before Feb 10 20:00 UTCNo execution_stats; only processing_time available
Phase 1 (v0.97.5)Feb 10 20:00 – Feb 11 16:00 UTCAfter base64 pre-decode + GenVM v0.2.12
Phase 2 (v0.98.2)Feb 11 16:00 – Feb 11 23:50 UTCAfter event-loop fix + validator thrashing fix
Phase 3a (contract)Feb 11 23:50 – Feb 12 01:47 UTCAfter contract storage optimization (v0.99.0 studio)
Phase 3b (v0.99.1)Feb 12 01:47 UTC onwards+ Shared cross-contract cache + validator timeout

Caveats

  • execution_stats only available from v0.98.0 onwards (PR #1434)
  • Phase 0 data limited to processing_time; host I/O breakdown unavailable
  • Some Phase 3 txs processed with old code (already in pipeline when upgrade deployed)
  • Non-sharded campaigns (DB73, C70B, 5771) unaffected by contract upgrade — changes there reflect studio improvements only
  • Validator processing times available but validator count constant (5 validators throughout)
